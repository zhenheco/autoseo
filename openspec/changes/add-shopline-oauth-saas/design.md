## Context

The current integration proves that SHOPLINE Admin API calls work through a private app token stored in 1Password. That path is useful for internal testing and high-trust CLI work, but it does not scale to ordinary merchants because customers will not reliably grant admin accounts or manually transfer API tokens.

The product direction is therefore:

- Public audit first, requiring only a store URL.
- OAuth app authorization second, requiring only a shop handle and one customer approval action.
- SaaS UI later, backed by the same token and operation modules used by CLI.

## Key Decisions

### Authorization Model

The primary customer flow is a dedicated SHOPLINE customized/customer app install. The app install URL is generated by 1waySEO, the customer approves scopes in SHOPLINE, and callback exchange happens server-side.

Private app tokens remain supported only for internal/trusted stores through 1Password-backed CLI commands.

### Token Storage

Customer OAuth tokens are stored in application storage associated with company, website, shop handle, scope, status, and audit metadata. Tokens must not be returned to the browser after exchange. Server-side callers resolve tokens by company and website ownership.

1Password remains the source of truth for platform app secrets and internal private app tokens. It is not the long-term storage layer for every customer SaaS OAuth token.

### Operation Layer

SHOPLINE operations should be exposed through a deep module with small interfaces:

- connection verification
- product SEO read/write
- page/content SEO read/write
- sitemap/public audit discovery
- operation dry-run and apply modes

CLI commands and SaaS routes should call this shared operation layer instead of duplicating API behavior.

### Customer Trust

The product must clearly state that 1waySEO does not need SHOPLINE passwords, admin accounts, orders, customers, payment, checkout, cart, or discount permissions for SEO work.

### Audit Before Write

Write operations should start with dry-run output and explicit apply commands. The first SaaS version may support assisted operation from CLI while UI catches up.

## Risks

- SHOPLINE customized app review/availability may block OAuth installs until app configuration is approved for the target stores.
- Scope names and Admin API payload shapes can differ from assumptions; tests must include real payload fixtures.
- Customer tokens must be encrypted and never logged.
